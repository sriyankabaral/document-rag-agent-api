# Document RAG Agent API

Backend-only FastAPI system for document ingestion, retrieval-augmented
generation, conversational memory, and interview booking.

## Required APIs

### 1. Document Ingestion

`POST /documents/upload`

Accepts PDF or TXT files, extracts text, chunks it, generates embeddings,
stores vectors and chunk payloads in Qdrant, and saves document metadata in
SQLite.

Supported chunking methods:

- `recursive` (default)
- `custom` fixed-size windows
- `sentence` (existing optional strategy)

Supported embedding models:

- `sentence-transformers/all-MiniLM-L6-v2` (default)
- `BAAI/bge-small-en-v1.5`

### 2. Agentic RAG

`POST /agent/query`

Uses LangGraph and ChatOllama with two tools:

- Qdrant document retrieval
- Interview booking through the centralized SQLite/SMTP booking service

Redis stores temporary user and assistant turns by `session_id`. Deterministic
intent routing keeps memory questions, document retrieval, and booking flows
separate.

## Supporting Endpoints

- `GET /health` - service health check
- `POST /documents/search` - retrieval-only Qdrant search
- `POST /book-interview` - direct interview booking API
- Swagger UI - `http://127.0.0.1:8000/docs`

## Technology Stack

- FastAPI and Uvicorn
- LangGraph, LangChain Core, and LangChain Ollama
- Ollama (`llama3.2` by default)
- Qdrant vector database
- Redis conversational memory
- SQLite and SQLAlchemy
- Sentence Transformers
- SMTP via Python standard library
- Docker Compose for Qdrant and Redis

Production retrieval filters Qdrant search by `embedding_model` to avoid
comparing vectors generated by different embedding models.

## Restrictions Followed

- No UI or Streamlit
- No FAISS
- No Chroma
- No RetrievalQA chain
- Redis is not used as the permanent booking database

## Setup

Create and activate a virtual environment:

```powershell
python -m venv venv
venv\Scripts\Activate.ps1
```

Install dependencies:

```powershell
pip install -r requirements.txt
```

Create a root `.env` file based on `.env.example`:

```powershell
Copy-Item .env.example .env
```

Important settings:

```env
OLLAMA_BASE_URL=http://localhost:11434
OLLAMA_MODEL=llama3.2
QDRANT_URL=http://localhost:6333
COLLECTION_NAME=document_chunks
REDIS_URL=redis://localhost:6379/0
REDIS_CONVERSATION_TTL_SECONDS=86400
REDIS_HISTORY_LIMIT=20
SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_EMAIL=your_sender_email@example.com
SMTP_PASSWORD=your_email_app_password
SMTP_TO_EMAIL=admin@example.com
SMTP_USE_TLS=true
```

The real `.env`, `app.db`, uploaded documents, evaluation documents, and
virtual environment are ignored by Git.

## Run Infrastructure

Open Docker Desktop, then start Qdrant and Redis:

```powershell
docker compose up -d
docker compose ps
```

Install and start the local Ollama model:

```powershell
ollama pull llama3.2
ollama list
```

Run FastAPI:

```powershell
uvicorn app.main:app --reload
```

## Test Document Upload

Open Swagger and execute `POST /documents/upload` with:

- `file`: a `.pdf` or `.txt` file
- `chunking_method`: `recursive`, `custom`, or `sentence`
- `embedding_model`: one of the supported models

Uploaded files are saved under `uploads/`. Their chunks and vectors are stored
in the Qdrant `document_chunks` collection, while metadata is stored in
`app.db`.

## Test Agentic RAG

Upload a document first, then call `POST /agent/query`:

```json
{
  "query": "What AI projects are mentioned in Sriyanka's resume?",
  "session_id": "candidate-123",
  "top_k": 5,
  "embedding_model": "sentence-transformers/all-MiniLM-L6-v2",
  "llm_model": "llama3.2"
}
```

Reuse the same `session_id` for follow-up turns. `conversation_id` is accepted
as an alias.

## Verify Redis Memory

First request:

```json
{
  "query": "My name is Sriyanka and my email is sriyanka.test@example.com.",
  "session_id": "memory-demo",
  "llm_model": "llama3.2"
}
```

Second request with the same session:

```json
{
  "query": "What is my email?",
  "session_id": "memory-demo",
  "llm_model": "llama3.2"
}
```

Expected response: the email is recalled from Redis,
`retrieved_context_count` is `0`, `sources` is `[]`, and `booking` is `null`.

Inspect memory directly:

```powershell
docker exec document_rag_redis redis-cli LRANGE agent:conversation:memory-demo 0 -1
```

## Test Interview Booking

Agent booking:

```json
{
  "query": "Book an interview for Test Candidate, candidate@example.com, on 2026-06-25 at 14:30.",
  "session_id": "booking-demo",
  "llm_model": "llama3.2"
}
```

Agent candidate notification is controlled in `app/config.py`:

```python
AGENT_NOTIFY_CANDIDATE_BY_DEFAULT = True
```

The direct `POST /book-interview` endpoint also supports an explicit
`notify_candidate` field. SQLite always stores the booking before SMTP is
attempted; email failures are recorded without deleting the booking.

## Retrieval Evaluation

Place local test files here (their contents remain ignored by Git):

```text
evaluation_data/resume.pdf
evaluation_data/it_company_regulations.txt
```

Ensure Qdrant is running, then execute:

```powershell
python scripts/evaluate_retrieval.py `
  --resume-path evaluation_data/resume.pdf `
  --regulations-path evaluation_data/it_company_regulations.txt
```

To include Ollama generation and measured end-to-end RAG latency:

```powershell
python scripts/evaluate_retrieval.py `
  --resume-path evaluation_data/resume.pdf `
  --regulations-path evaluation_data/it_company_regulations.txt `
  --include-llm `
  --llm-model llama3.2
```

The script compares:

- Recursive and custom fixed-size chunking
- MiniLM and BGE-small embeddings
- Qdrant default HNSW and exact vector search

It creates separate collections:

- `eval_recursive_minilm`
- `eval_recursive_bge`
- `eval_custom_minilm`
- `eval_custom_bge`

Embedding, Qdrant search, total retrieval, optional LLM, end-to-end latency,
and source accuracy are written to:

```text
reports/retrieval_evaluation_results.csv
```

Measured results, observations, assumptions, and limitations are documented
in:

```text
reports/FINDINGS.md
```

The included evaluation retrieved the expected document in all 32 tests. See
the report for latency comparisons and why the small dataset does not support
a universal winning configuration.
